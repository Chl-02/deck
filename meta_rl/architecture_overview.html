HEV Meta-EMS · Meta-RL Testbed

모델 아키텍처 개요

양산형 map/rule 하이브리드 에너지 관리(EMS) 위에 Meta-RL을 제어기 대체가 아니라 경계가 정해진 보정(calibration) 어댑터로 얹은 구조를, 처음 보는 사람도 이해할 수 있게 정리한 문서.

한 문장 요약 — 공장에서 튜닝된 EMS 제어기가 운전대(엔진 on/off·파워 분배)를 잡고, Meta-RL은 조수석에서 9개 다이얼(calibration)만 정해진 폭 안에서 미세조정하며, Confidence Gate(안전 감독관)가 낯선 상황에서 그 손을 붙잡아 규칙으로 되돌린다.

1. 큰 그림과 권한 경계

이 프로젝트는 production-like map/rule HEV EMS 위에 Meta-RL을 올려, 처음 보는 주행 조건(unseen / OOD / 날씨 / 스트레스)에서 SOC 조절 tail risk를 줄일 수 있는지 평가하는 논문용 testbed다.

가장 중요한 설계 원칙은 권한 경계(authority boundary)다:

Meta-RL   →  경계가 정해진 9-knob calibration residual / knob  (만 조정)
Map/Rule  →  엔진 상태, EMS 모드, 파워 분배, 최종 command  (를 결정)
절대 이렇게 해석하지 않는다: Meta-RL이 엔진/모터 파워, 토크 분배, 엔진 command를 직접 최적화한다 · 이 모델이 정확한 양산 ECU calibration이다 · 절대 연비 비교가 가능하다. (모델은 FASTSim 기반 backward, power-domain 단순화 모델이다.)

3. 제어 흐름 — 한 스텝에서 일어나는 일

1. 관측
현재 SOC, 속도, 파워 요구, 최근 이력을 읽는다.
2. context 추론
최근 이력 → 잠재 벡터 z ("이건 어떤 주행인가").
3. 어댑터 출력
정책이 9-knob 조정(bounded)을 낸다.
4. gate 감쇠
낯섦/스트레스면 조정을 줄여 base로 복귀.
5. EMS 결정
보정된 다이얼로 엔진 on/off·파워 분배 결정.
6. 물리 적분
SOC·연료·배터리 갱신, 보상 계산.

어댑터는 매 스텝이 아니라 adapter_update_period(약 10초)마다 갱신되고, 각 다이얼은 rate-limit으로 부드럽게만 움직인다 — 갑자기 튀지 않는다.

4. 9개 calibration 다이얼

Meta-RL이 만질 수 있는 것은 오직 이 9개뿐이며, 모두 공장 캘리브레이션에 더하는 조정량(delta)이고 전부 bounded다.

#다이얼물리적으로 무엇을 바꾸나범위(bound)
1ΔP_eng_on엔진 켜는 파워 바이어스 — 배터리 충전 위해 엔진을 더/덜 밀어줌residual ±1 kW, 최종 ±3 kW
2ΔSOC_refEMS가 맞추려는 목표 SOC 기준값 이동 (↑=배터리 더 꽉 채움)residual ±0.01, 최종 ±0.02
3SOC_recovery_gainSOC가 목표보다 낮을 때 회복하는 세기 (↑=빠르게 충전)[−0.4, +0.6]
4engine_on_SOC_sens엔진-on 결정이 SOC 부족에 반응하는 민감도[−0.4, +0.6]
5P_hysteresis엔진 on/off 임계 주변 불감대 — 채터링 방지 (↑=스위칭↓)적용 [2, 15] kW
6min_on_time엔진이 켜지면 최소 유지 시간 (↑=스위칭↓)Δ[−30,+30]s, 적용 [5,90]s
7min_off_time엔진이 꺼지면 최소 재시동 금지 시간Δ[−5,+20]s, 적용 [5,90]s
8battery_power_margin배터리 파워 한계를 얼마나 쓸지 (↓=여유 남김·보호)[0.75, 1.0]
9regen_margin허용 회생제동 파워 (↓=회생 덜 씀·눈길 대응)[0.5, 1.0]

기능별 4그룹으로 묶으면

① 세트포인트 (어디에 맞출까)   : 1 ΔP_eng_on, 2 ΔSOC_ref
② 반응 세기 (얼마나 세게)      : 3 recovery_gain, 4 SOC_sensitivity
③ 스위칭 동작 (엔진 껐다 켰다) : 5 hysteresis, 6 min_on, 7 min_off
④ 파워 여유 (배터리/회생 한계) : 8 battery_margin, 9 regen_margin

SOC tail risk는 주로 ①②로, 엔진 스위칭(가드레일)은 ③으로, 날씨·배터리 derating 같은 OOD는 ④로 대응한다. Phase A 정적 분석에서 #6 min_on_time이 가장 강력한 지렛대였다.

5. base + residual 구조 핵심

최종 다이얼 값은 항상 기준값(base) + 정책이 더하는 보정(residual)으로 조립된다. 이 구조가 이 프로젝트 연구의 중심이다.

최종 calibration  =  base(기준)  +  gate · residual(정책 보정)

base를 누가 주느냐가 성능을 좌우한다. 여기서 "capture %"는 adaptive baseline에서 정적 oracle 상한까지의 개선 여지를 몇 % 잡았는가를 뜻한다(높을수록 좋음).

버전base 기준base 성능(capture)낮은 confidence에서 fallback
V3~V5hand-coded adaptive rule (반응형)~14%adaptive rule로 복귀
V6 (P3, 현재)학습된 fitted rule (oracle 회귀)~32%fitted rule로 복귀
P3(residual-on-fitted-base)의 아이디어 — 9개 다이얼 전부를 fitted rule(32%)이 base로 주고, 정책은 그 위에 bounded residual(정규화 ±0.3)만 더한다. gate가 residual을 억제해도 fallback이 강한 fitted rule이라, 낯선 조건에서도 성능 바닥이 높다. 정책은 처음부터 32%에서 출발해 한계 개선만 학습하면 된다.
P3:   final = fitted_rule(obs) + gate · bounded_residual   // confidence 0 → fitted rule로 정확 복귀

6. Confidence Gate — 안전 감독관

gate는 "지금 이 조정을 믿어도 되나"를 판단해 residual을 감쇠한다. 하지만 실제로 재는 것은 "유능함"이 아니라 두 가지 대리지표다:

채널 1 — 낯섦
11개 관측 feature가 학습 분포 중앙값에서 얼마나 벗어났나 (z-score). "이 입력이 특이한가?"
채널 2 — 스트레스
SOC 위험, 배터리 한계 사용률, 최근 constraint 위반. "지금 SOC/배터리가 힘든가?"
긍정 결과 — gate는 OOD tail risk를 +6.3pt 개선한다(2026-07-17 ablation). 안전장치가 성능도 개선하는, 그 자체로 논문 기여 후보.
근본 한계 — gate가 억제해야 하는지는 "정책이 이 영역을 학습했는가"에 달렸는데, gate는 "특이하거나 힘든가"만 본다. 이 둘은 다른 축이다. 예컨대 학습한 날씨는 특이·힘들어 보이지만 정책은 유능한데, gate가 좋은 적응까지 깎는다. 게다가 숨은 축(마찰·accessory·배터리 derating)은 관측에서 격리돼 있어 gate에 애초에 안 보인다. → 이래서 전역 attenuation floor는 net-negative였고(P1), 강한 fallback을 주는 P3나 판별형 gate가 옳은 방향이다.

7. "Meta" 학습 — 상황을 추론하는 법

정책은 두 부분으로 된 probabilistic PEARL-style context encoder + SAC다:

  • Context encoder: 최근 주행 이력(support set)을 보고 "지금 어떤 종류의 주행인가"를 잠재 벡터 z로 추론한다. → 태스크마다 다르게 반응하는 원천.
  • SAC 정책: 관측 + z를 받아 9-knob 조정을 출력한다.

즉 "새 조건을 만나면 몇 스텝 안에 스스로 튜닝을 맞춘다"가 meta의 의미다. 학습 태스크는 drive cycle + 숨은 축의 조합으로 매 에피소드 샘플링되고, 보상은 연료·SOC tracking·tail risk·배터리·스위칭·action penalty의 다항 합이다.

warm-start: 과거(V4/V5)에는 fitted rule을 모사하는 imitation pretraining을 썼지만, P3(V6)에서는 fitted rule이 구조적으로 base가 되므로 별도 pretraining 없이 정책이 residual≈0(=32%)에서 출발한다.

9. 주장 가능 범위와 한계

안전하게 주장 가능:

  • 경계가 정해진 task-aware calibration 어댑터가 권한 경계를 보존하면서 unseen/stress 조건의 SOC tail risk를 줄일 수 있다.
  • Confidence-gated baseline reversion이 OOD 개입을 보수적으로 만들고, OOD capture를 +6.3pt 개선한다.
  • rule이 구조적으로 못 만지는 오보정(ems-variant) 축에서 meta-RL이 우위를 보인다.

피해야 할 주장:

  • 큰 보편적 연비 개선 / 모든 baseline 압도 / 모든 split에서 constraint 감소 보장.
  • Meta-RL이 엔진·모터 파워 분배를 직접 최적화한다.
  • 강한 adaptive/fitted rule을 검증 없이 확실히 능가한다 — 현재 fitted rule(supervised, ~32%)이 여전히 강한 상한이다.
현재 정직한 위치 — "성능"보다 권한/성능 tradeoff, gate의 OOD 보호, 훈련 커버리지의 인과적 중요성(V5), supervised headroom을 RL이 어떻게 좁히는가(V6/P3)"가 이 testbed의 실질 기여다.
HEV Meta-EMS · Meta-RL Testbed — 아키텍처 개요. 정확한 수치와 최신 실험은 프로젝트 living ledger 및 원본 결과 CSV/JSON에서 검증한다.
권한 경계: Meta-RL → bounded 9-knob calibration residual/knob · Map/Rule EMS → engine state·mode·power split·final command.
